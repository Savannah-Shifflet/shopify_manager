## 1. Problem Statement

Every authorized reseller — regardless of niche — runs the same operational loop: find suppliers, negotiate relationships, import product catalogs, build out listings, and track pricing. Today this loop is entirely manual: spreadsheet wrangling, copy-pasted supplier emails, one-by-one product entry, and no automated price watching. The existing standalone MVP (`shopify_products/`) proved the concept but lives outside Shopify, requires a developer to configure, and cannot scale to a real customer base.

The gap: there is no Shopify-embedded tool built specifically for the authorized reseller workflow. General-purpose Shopify apps (Oberlo, DSers) target dropshippers. PIM tools (Akeneo, Salsify) target enterprise. SourceDesk targets the mid-market authorized reseller who sells $200–$10,000+ items from 3–20 brand suppliers and needs professional infrastructure without a $50K/yr software budget.

**Who:** Authorized resellers operating Shopify stores — every aspect of their sourcing, importing, Shopify management, and catalog organization runs through this app.

**What:** A Shopify-embedded (App Bridge) application replacing the standalone `shopify_products` MVP with a production-grade, multi-tenant platform.

**Why:** Manual operations don't scale. A reseller with 500 SKUs across 8 suppliers cannot manage price changes, content quality, and supplier outreach with spreadsheets and Gmail.

---

## 2. Evidence

- Existing `shopify_products/` MVP validates the core import and enrichment workflow but is a single-user standalone tool — not shippable as a product
- Authorized reseller forums and communities consistently surface the same four pain points: supplier discovery, catalog data entry, content quality, and price tracking
- No direct Shopify App Store competitor solves all four pain points for authorized resellers specifically
- The reseller workflow is category-agnostic — the same app serves hunting gear, luxury furniture, fitness equipment, and powersports dealers

---

## 3. Proposed Solution

SourceDesk extends the existing `shopify_products` MVP architecture into a fully embedded, multi-tenant Shopify application. The extension strategy:

- **Extend existing patterns** rather than replacing them: the enrichment pipeline, column-mapping import flow, and scraping infrastructure from the MVP carry forward
- **Add the missing layer**: multi-tenancy (`shopDomain` scoping on every query), Shopify App Bridge embedding, Shopify billing, and email OAuth
- **Four integrated modules** that work together in a single UI: Supplier CRM → Email Outreach → Catalog Import → AI Enrichment + Sync, with Price Monitoring running continuously in the background
- **AI is non-destructive throughout**: every AI output lands in staging fields; merchant reviews before anything reaches Shopify

The app lives entirely inside Shopify Admin as an embedded app. Merchants never leave their admin panel to use any feature.

---

## 4. Key Hypothesis

**Primary:** If we give authorized resellers a single embedded tool that handles their entire sourcing-to-listing workflow, they will pay $79–$299/month and reduce time-to-first-product-synced from days to under 30 minutes.

**Secondary:** AI-generated content that follows a merchant's own template and brand voice will be accepted (not rejected) >70% of the time after review, making the enrichment feature sticky enough to drive tier upgrades.

## 7. Users & Context

### Primary Persona — The Authorized Reseller

- **Who:** A Shopify store owner selling high-ticket items ($200–$10,000+ per unit) from brands they are an authorized dealer of. Categories: outdoor/hunting gear, luxury goods, fitness equipment, furniture, electronics, sporting goods, powersports.
- **Business model:** Apply to brands for dealer authorization → receive wholesale pricing → resell at MAP or above. Catalog spans 3–20+ supplier brands with hundreds to thousands of SKUs.
- **Jobs-to-be-done:**
  1. Find legitimate new suppliers for their niche without manual directory research
  2. Convert supplier spreadsheets and websites into Shopify listings without manual data entry
  3. Ensure product content is brand-consistent and professional without writing each description
  4. Know immediately when a supplier changes prices so they're never selling below cost or MAP
- **Pain points (daily):** Supplier discovery is manual and time-consuming. Cold outreach is inconsistent. Supplier product sheets require hours of data entry. Descriptions are copy-pasted from PDFs. Price changes require manual catalog auditing.
- **Technical comfort:** Moderate. Comfortable with SaaS tools. Will not edit Liquid theme files. Expects guided setup.
- **Usage pattern:** Several times per week. Heavy during catalog onboarding and supplier acquisition. Lighter for ongoing monitoring.

### Secondary Persona — The Store Manager / VA

- Delegated access to specific functions (product imports, email outreach) under the owner's account
- Needs clear, task-based workflows — no ambiguity about next action
- **Not a decision-maker** — cannot change pricing rules, billing, or core settings

### Non-Users
- Dropshippers (no supplier relationships, different model)
- Enterprise retailers (PIM tools handle their scale)
- Suppliers themselves (no supplier-facing portal at MVP)

## 8. Open Questions

1. **Onboarding blocking granularity** — Which specific features are blocked (banner shown) before onboarding completes? Full app lockout? Per-feature? Current code has `onboardingDone` flag on `MerchantConfig` but blocking logic is not specified.
2. **Notification delivery** — When a supplier replies or a price changes, how does the in-app notification reach the merchant? Polling? Shopify App Bridge notification API? `TRACKING_BASE_URL` env var suggests email pixel tracking exists — is Resend used for transactional alerts? (Not in current `.env.example`)
3. **Audit log spec** — Phase 4 includes an audit log but there is no `AuditLog` model in the schema and no spec in Section 7. What actions are logged? Where is it stored?
4. **Extensions/content-blocks** — The Theme App Extension (`extensions/content-blocks/`) does not yet exist. This is the Liquid block for rendering metafield content as tabs/accordions in storefronts. When is this built? Phase 3 includes it.
5. **IMAP vs Gmail/Graph API for reply detection** — Spec says "IMAP sync job" but Gmail API and Microsoft Graph API (REST) are the OAuth-based read mechanisms. Is IMAP actually used or is it Gmail API polling? This affects the `email-sync.job.ts` implementation.
6. **`DescriptionTemplate` storage** — v1.0 PRD shows a dedicated `DescriptionTemplate` model. Actual schema stores `contentTemplate` as JSON inside `MerchantConfig`. Multiple templates per merchant (as specified) require the separate model — is this planned schema migration?
7. **Scraper rate limiting and politeness** — No spec for how frequently catalog scrapes or price monitor scrapes hit the same domain. What is the minimum delay between requests to the same host?
8. **Email tracking pixel** — `TRACKING_BASE_URL` is in `.env.example`. How does open tracking work? Is there a `/track/open/:messageId` route? This is not in the route directory.
9. **MAP enforcement UX** — When `mapHardBlock` is true in `StoreSettings` (not present in actual schema — see data model section), does the app block the sync entirely or just alert? Needs a clear error state spec.
10. **Multi-contact outreach** — Supplier contacts are stored as a JSON array. When enrolling a supplier in an email sequence, which contact is the sequence sent to? Is there contact selection UI?
11. **Shopify Billing + Railway deployment** — The app needs to run two processes (web + worker). Railway supports this via a `Procfile` or service config. Is there a `Procfile` in the repo? (Not verified.)
12. **Onboarding resume state** — `MerchantConfig.onboardingStep` tracks progress. What happens if a merchant partially completes onboarding, then the app updates and the step count changes?
13. **`SHOPIFY_STORE_DOMAIN` in multi-tenant context** — The `.env.example` includes `SHOPIFY_STORE_DOMAIN`, which is a single-store variable. In a multi-tenant app, `shopDomain` comes from the session — this env var appears to be a legacy artifact from the standalone MVP. Needs removal or clarification.

### Request Flow

1. Merchant opens app → App Bridge provides session token (JWT)
2. Every Remix loader/action calls `authenticate.admin(request)` → validates JWT, returns scoped GraphQL client and `shopDomain`
3. All DB queries scoped by `shopDomain` extracted from authenticated session — enforced at service layer
4. Heavy operations (scraping, enrichment, sync) enqueued to BullMQ via `app/jobs/queues.ts` — never executed inline
5. UI polls job status via lightweight loaders (no WebSockets at MVP scale)

### Multi-Tenancy

- `shopDomain` column on every Prisma model (indexed)
- All service-layer functions require `shopDomain` as mandatory parameter
- Never pass a Prisma query without a `shopDomain` filter — code review gate
- Future: PostgreSQL RLS as defense-in-depth layer (SQLite dev, PostgreSQL prod)

### AI Non-Destructive Pattern

- All AI outputs land in staging fields: `aiTitle`, `aiDescription`, `aiTags`, `aiAttributes`
- `applyAiAcceptance()` in `app/services/ai-acceptance.service.ts` is the **only** place staging fields are promoted to main fields
- `ACCEPTANCE_MAP` in the same file defines all field mappings
- Rejecting AI: staging fields cleared, `enrichStatus` reset to `NOT_STARTED`

### Content Architecture (Metafields)

- Structured content (tabs, accordions, specs) → Shopify metafields via `metafieldsSet` GraphQL mutation
- Namespace: `custom.*` for merchant-visible fields (connectable in Theme Editor)
- Namespace: `$app:*` for internal app state (enrichment metadata, confidence scores)
- Theme App Extension (Liquid block): ⚠️ **must be created** at `extensions/content-blocks/` in Phase 3
- `body_html` reserved for plain prose narrative only

| # | Decision | Rationale | Date |
|---|---|---|---|
| 1 | Shopify App Bridge embedded app over standalone | Must live inside Shopify Admin for merchant convenience and App Store distribution | Pre-v1.0 |
| 2 | Remix (React Router v7) as full-stack framework | Official Shopify app framework; loaders/actions map cleanly to Shopify auth pattern | Pre-v1.0 |
| 3 | BullMQ + Redis for async jobs | All heavy operations (scraping, AI, sync) are async; BullMQ provides retries, dead-letter, repeatable jobs | Pre-v1.0 |
| 4 | AI outputs land in staging fields only — never auto-applied | Merchant trust is non-negotiable; AI errors on published listings are business-damaging | Pre-v1.0 |
| 5 | Metafields over body_html for structured content | Allows Theme Editor connection, separates data from presentation, enables storefront flexibility | Pre-v1.0 |
| 6 | `shopDomain` required on every service function | Prevents cross-tenant leakage at the code layer before RLS can be added | Pre-v1.0 |
| 7 | OAuth email (Gmail/Outlook) from merchant's own account | Cold email from app infrastructure risks spam reputation; merchant's own account is the CAN-SPAM-compliant path | Pre-v1.0 |
| 8 | SQLite in dev, PostgreSQL in production | SQLite zero-config for local dev; production needs RLS, connection pooling, and JSON operators | v2.0 review |
| 9 | `EmailAccount` as dedicated model (not embedded in StoreSettings) | Clean separation of concerns; each merchant may eventually connect multiple accounts | v2.0 review |
| 10 | `PriceMonitorConfig` as separate model from `Supplier` | One supplier may eventually have multiple monitored URLs; separate model enables this without schema churn | v2.0 review |
| 11 | Single-file `app.onboarding.tsx` (not directory) | Onboarding is a step-managed single page with state; Remix nested route directory not needed at current scope | v2.0 review |
| 12 | `ENCRYPTION_KEY` over `APP_SECRET` | More descriptive; clearly signals it is a symmetric key, not a generic secret | v2.0 review |
| 13 | `DescriptionTemplate` needs dedicated model (not MerchantConfig JSON) | Multi-template support (different templates per product type) was specified in v1.0; single JSON field in MerchantConfig does not support this — migration required | v2.0 review |

## 14. Security & Configuration

### Authentication
- **Shopify Token Exchange** (post-Feb 2024 default) — no legacy cookie sessions
- Every route calls `authenticate.admin(request)` — invalid sessions auto-redirect to OAuth
- Per-merchant OAuth tokens for Gmail/Outlook stored encrypted in `EmailAccount` model using `ENCRYPTION_KEY`
- HMAC verification on all incoming Shopify webhooks (handled in `webhooks.tsx`)

### Multi-Tenancy Enforcement
- `shopDomain` column on every Prisma model — indexed
- All service-layer functions require `shopDomain` as a mandatory parameter
- Never pass a Prisma query without a `shopDomain` filter — enforced in code review
- Future: PostgreSQL RLS as defense-in-depth layer

### Secrets Management
- Never log OAuth tokens, API keys, or `shopDomain`-scoped data
- Stored OAuth tokens encrypted at rest using `ENCRYPTION_KEY` (32 bytes, hex-encoded)
- No secrets in source control — `.env` in `.gitignore`

### Shopify App Store Requirements
- GDPR webhook handlers: `customers/data_request`, `customers/redact`, `shop/redact` — implemented in `webhooks.tsx`
- Polaris used throughout (no competing UI library)
- App Bridge token exchange only (no legacy OAuth redirect flow)
- All billing via Shopify `appSubscriptionCreate` (no external payment collection)
- Cold email sends from merchant's own OAuth-connected account (CAN-SPAM compliance)

## 16. Risks & Mitigations

| Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|
| Supplier website scraping blocked by anti-bot | High | Medium | ScraperAPI managed proxy; Playwright for JS-rendered pages; graceful degradation (notify merchant, suggest manual import) |
| Gmail/Outlook OAuth token expiration causing silent email failures | Medium | High | Background token refresh before `expiresAt`; in-app reconnect prompt when token invalid; never fail silently on email operations |
| Cross-tenant data leak from missing `shopDomain` filter | Low | Critical | `shopDomain` required param enforced at service layer; code review gate on all Prisma queries; future RLS layer |
| Shopify App Store rejection for cold email feature | Medium | High | Cold email sends from merchant's own OAuth account (not app infrastructure); document CAN-SPAM compliance; consult Shopify partner support pre-submission |
| AI description quality inconsistency on sparse product data | Medium | Medium | Confidence scoring with every generation; sparse data flags in prompt; mandatory merchant review before acceptance; never auto-apply |
| SQLite → PostgreSQL migration breaks schema assumptions | Medium | High | Use Prisma migrations throughout dev so production migration is additive; test with PostgreSQL in CI before launch |
| Theme App Extension not built before sync goes live | Medium | Medium | Metafields are written correctly regardless; the Extension is a rendering layer only — content is not lost if Extension is delayed |
| `SHOPIFY_STORE_DOMAIN` env var causing confusion in multi-tenant context | Low | Medium | Remove or clearly document as dev-only/legacy artifact; `shopDomain` always comes from authenticated session at runtime |
| Multi-template support gap (DescriptionTemplate model missing) | High | Medium | Migration required before Phase 3 enrichment feature is complete; do not ship template builder against `MerchantConfig.contentTemplate` JSON only |
